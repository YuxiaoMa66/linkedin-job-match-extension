# LinkedIn Job Match 0.1.1 产品规格

## 版本目标

`0.1.1` 聚焦于把当前“单一评分逻辑”升级成“可配置的分析系统”，并同时补足三块基础能力：

1. 分析偏好与权重可配置
2. 错误提示对普通用户更友好
3. 性能与稳定性可被观测和排查

这次版本不追求大范围重做 UI，而是在现有架构上补一层更完整的产品能力，确保后续继续迭代时不会因为缓存、设置和结果解释方式不一致而返工。

## 0.1.1 范围

### 必做范围

- 新增 4 个分析偏好预设
- 新增高级权重设置
- 新增 Prompt Tuning 配置层
- 调整缓存键，避免不同偏好/模型共用旧结果
- 新增错误分类与用户友好提示
- 新增本地性能监控与调试信息

### 不纳入 0.1.1 的范围

- 云端账户同步
- 多人协作或服务端存储
- A/B 实验系统
- 复杂的可视化图表面板
- Chrome Web Store 上架流程本身

## 用户问题定义

当前版本存在以下体验问题：

1. 同一个岗位只会基于一套固定权重打分，无法表达“我更看重经验”或“我更看重 sponsorship”的个人偏好。
2. 切换模型、Prompt 或偏好后，缓存仍可能命中旧结果，造成“分数没变但设置明明变了”的困惑。
3. 错误提示过于技术化，用户不知道应该修改 API key、模型名、页面状态还是稍后重试。
4. 缺少性能指标，难以判断慢在 JD 提取、LLM 调用还是缓存命中失败。

## 目标用户

- 正在使用 LinkedIn 进行岗位筛选的个人用户
- 关注荷兰岗位与 `KM` sponsorship 的用户
- 会自己配置 LLM provider 的进阶用户
- 对评分逻辑有个性化偏好的用户

## 核心设计

### 一、分析偏好预设

0.1.1 提供 4 个预设偏好，默认值为 `平衡评估`。

#### 1. 严格匹配 `strict`

适合用户：

- 优先筛掉明显不匹配岗位
- 更看重硬技能、明确年限、必需语言

设计原则：

- 提高技能与经验年限权重
- 对缺失关键要求更敏感
- 降低“潜力型匹配”的宽容度

#### 2. 平衡评估 `balanced`

适合用户：

- 希望系统默认给出较稳妥、通用的匹配结果

设计原则：

- 维持当前版本最接近的综合评分逻辑
- 兼顾技能、职责、经验、语言和 sponsorship

#### 3. 潜力优先 `potential`

适合用户：

- 愿意尝试成长型岗位
- 不希望系统因为年限不足而过度压低分数

设计原则：

- 提高职责覆盖与可迁移技能权重
- 下调经验年限的影响
- 鼓励“能力可迁移”的岗位进入候选

#### 4. 签证优先 `sponsorship-first`

适合用户：

- 主要关注荷兰岗位
- 首要目标是筛出更有 sponsorship 可能性的岗位

设计原则：

- 显著提高 sponsorship 权重
- 降低纯背景项的影响
- 更早突出 `KM`/赞助信号

### 二、预设权重

下表为在“荷兰岗位且 sponsorship 项可用”时的默认权重：

| 维度 | strict | balanced | potential | sponsorship-first |
| --- | ---: | ---: | ---: | ---: |
| Skills & Experience Relevance | 0.32 | 0.30 | 0.32 | 0.24 |
| Responsibility Coverage | 0.24 | 0.25 | 0.28 | 0.18 |
| Years of Experience | 0.20 | 0.15 | 0.08 | 0.10 |
| Education & Certifications | 0.10 | 0.15 | 0.12 | 0.08 |
| Language & Location | 0.10 | 0.10 | 0.12 | 0.10 |
| Sponsorship Fit | 0.04 | 0.05 | 0.08 | 0.30 |

#### 非荷兰岗位的处理

如果岗位不在荷兰，或 sponsorship 数据不可用：

- `Sponsorship Fit` 从 breakdown 中移除
- 其余维度按比例重新归一化至 `100%`
- 当前使用的偏好名称保持不变，但 UI 中要提示“当前岗位未应用 sponsorship 项”

### 三、高级权重设置

在 `Settings` 中新增 `Analysis preference` 区块：

- 预设选项卡：`Strict` / `Balanced` / `Potential` / `Sponsorship-first`
- 高级设置折叠面板：`Advanced weights`

高级设置规则：

- 每个维度使用 `0-100` 的滑杆或数字输入框
- 界面中显示总和
- 用户点击 `Normalize` 时自动归一化为 `100%`
- 用户点击 `Reset to preset` 时恢复当前预设默认值
- 用户改动任何一个权重后，当前模式自动标记为 `Custom`

推荐交互：

- 普通用户只需要选预设
- 高级用户展开后可精调
- 不把 6 个权重默认全部摊开，避免设置页过重

### 四、Prompt Tuning 设计

Prompt Tuning 采用“两层结构”：

#### 第一层：风格模板

与 4 个偏好联动，但可以单独调整。建议增加字段：

- `promptTuningMode`
- 可选值：`strict` / `balanced` / `potential` / `sponsorship-first`

它影响：

- 提示词对“必须项”的强调程度
- 对经验年限的容忍度
- 对 sponsorship 的关注程度
- strengths / gaps 的描述风格

#### 第二层：附加说明

新增可选字段：

- `additionalPromptInstructions`

用途：

- 让高级用户补充一段自己的偏好说明
- 例如“更看重数据分析与 SQL”“请弱化教育背景影响”

约束：

- 长度上限建议 `500` 字符
- 只作为 user prompt 追加说明，不允许覆盖系统 JSON 输出格式

### 五、缓存与重新分析

当前缓存键只包含：

- `jobId + resumeHash`

0.1.1 必须改成包含分析配置的版本，否则不同偏好共用缓存会造成结果错误。

#### 新缓存键设计

新增内部概念：

- `scoringProfileHash`
- `modelKey`
- `promptVersion`

推荐逻辑：

- `modelKey = provider + "::" + normalizedBaseUrl + "::" + modelId`
- `scoringProfileHash` 由以下内容稳定哈希生成：
  - preset id
  - normalized weights
  - promptTuningMode
  - additionalPromptInstructions
- `promptVersion` 由 prompt 模板代码手动维护，例如 `v3`

推荐缓存键：

`match_result_v3_${resumeHash}_${scoringProfileHash}_${modelKeyHash}_${jobId}`

#### 缓存读取规则

- 只有当 `jobId`、`resumeHash`、`scoringProfileHash`、`modelKeyHash`、`promptVersion` 全部一致时才命中
- 如果用户点击 `Re-analyze`，强制跳过缓存
- 如果用户切换预设或修改权重，应立即视为新的分析上下文

#### 缓存 summary 需要新增的字段

- `analysisPreset`
- `isCustomProfile`
- `weightsSnapshot`
- `promptTuningMode`
- `provider`
- `modelId`
- `promptVersion`
- `timing.totalMs`
- `timing.llmMs`
- `timing.extractMs`
- `cacheVersion`

### 六、错误处理

0.1.1 的错误提示需要从“技术报错”改成“用户可操作提示”。

#### 错误分类

建议统一到以下类型：

- `CONFIG_MISSING`
- `AUTH_FAILED`
- `MODEL_NOT_FOUND`
- `RATE_LIMITED`
- `NETWORK_FAILED`
- `LINKEDIN_CONTEXT_UNAVAILABLE`
- `JD_EXTRACTION_FAILED`
- `RESUME_PARSE_FAILED`
- `MODEL_OUTPUT_INVALID`
- `CACHE_MISMATCH`
- `UNKNOWN_ERROR`

#### 用户提示原则

- 标题说明“发生了什么”
- 正文说明“为什么可能会这样”
- 动作说明“下一步建议怎么做”

#### 示例

`MODEL_NOT_FOUND`

- 标题：模型不可用
- 说明：当前 provider 返回了模型不存在或无法访问
- 建议：检查 model id，或先点击 `Test connection`

`LINKEDIN_CONTEXT_UNAVAILABLE`

- 标题：当前页面还没有读取到岗位信息
- 说明：LinkedIn 页面可能尚未完成渲染，或当前标签页不是职位页
- 建议：刷新职位页后重试，或切回具体岗位页

`MODEL_OUTPUT_INVALID`

- 标题：模型已响应，但结果格式不完整
- 说明：当前模型没有按预期返回完整 JSON
- 建议：重试一次，或切换到更稳定的模型

### 七、性能监控

0.1.1 引入“仅本地”的轻量性能监控，不做外发遥测。

#### 监控目标

- 判断慢在哪里
- 判断坏结果来自哪里
- 帮助调试 provider 稳定性

#### 每次分析记录的字段

- `analysisId`
- `startedAt`
- `finishedAt`
- `totalMs`
- `extractMs`
- `llmMs`
- `parseMs`
- `repairMs`
- `cacheHit`
- `usedRetry`
- `provider`
- `modelId`
- `analysisPreset`
- `isCustomProfile`
- `errorType`
- `resultScore`

#### 数据保存策略

- 保存在 `chrome.storage.local`
- 仅保留最近 `100` 次分析记录
- 默认开启
- 后续可在设置中增加 `Diagnostics` 开关

#### UI 呈现

先不做复杂图表，只在 `Debug` 折叠区显示：

- 本次总耗时
- JD 提取耗时
- LLM 调用耗时
- 是否命中缓存
- 是否发生重试
- 当前 preset / custom profile 信息

### 八、设置结构调整

建议在现有 `Settings` 里新增以下区块：

#### Analysis preference

- `Preset`
- `Prompt tuning`
- `Advanced weights`
- `Additional instructions`

#### Diagnostics

- `Show analysis timings`
- `Keep recent run logs`

Provider 设置仍保留原来的结构，不与分析偏好混在一起。

## 迁移方案

### 配置迁移

- 老用户如果没有 `analysisPreset`，默认赋值为 `balanced`
- 老用户如果没有 `promptTuningMode`，默认跟随 `analysisPreset`
- 老用户如果没有 `customWeights`，默认使用空对象

### 缓存迁移

- 保留读取老缓存的能力没有意义，因为评分逻辑已变化
- 启动时直接清理旧 `match_result_v2_` 数据
- 从 `v3` 开始重新累计缓存

## 验收标准

### 功能验收

- 用户可在 4 个预设之间切换，并得到不同结果
- 用户修改权重后，模式显示为 `Custom`
- 切换预设或模型后，不会错误命中旧缓存
- `Re-analyze` 始终跳过缓存
- 非荷兰岗位不会错误显示 sponsorship 权重

### 体验验收

- 出错时用户能看懂“问题是什么”和“下一步怎么做”
- `Debug` 信息能够解释“为什么这次慢”
- 设置页没有明显复杂到影响基础使用

## 建议开发顺序

1. 先做 constants / config / cache key 升级
2. 再做 match-engine 的权重与 prompt tuning
3. 再做 sidepanel 设置 UI
4. 最后补错误映射与性能监控
